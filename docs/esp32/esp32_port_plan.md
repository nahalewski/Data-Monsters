# Core ESP32 Frontend Port Package

## 1) Repo audit
Audit source: `https://github.com/nahalewski/Core` (commit `681ee1b`, inspected on 2026-04-12).

### UI-related assets and interfaces
- `face_ui/os_launcher.html` contains a circular launcher, theme system, app order persistence, long-press jiggle/reorder behavior, and app grid pages.
- `face_ui/index.html` is the primary Core app surface with face state rendering, PTT controls, chat dock, and whiteboard overlay.
- Additional app pages (`learning.html`, `music.html`, `settings.html`, `gallery.html`, `clock.html`, `routines.html`, `ink.html`, etc.) implement the multi-app flow.
- `core_mobile/` is a Flutter target with image assets and native wrappers for mobile/desktop platforms.

### App states and navigation model
- Runtime state polling is driven by `/status` and includes `current_state`, `status_text`, transcript, setup flags, and settings.
- Core states inferred from UI/backend are: `happy`, `listening`, `thinking`, `speaking`.
- Navigation model is launcher-first: app tiles route to dedicated app screens; status bar + page dots reinforce circular launcher metaphor.

### Circular UI metaphors
- Circular viewport, clipped screen, radial visual language, central "Core" presence, and hub-style launcher are core identity elements.
- Jiggle mode + icon reorder are strongly iOS-like but wrapped in a circular vessel.

### Audio input/output behavior
- Audio in: `audio_input.py` records 16k mono, push-to-talk semantics (`/ptt/start`, `/ptt/stop`).
- Audio out: `audio_output.py` calls ElevenLabs, streams mp3, and sequentially plays clips (`afplay`) with interruption support.

### Memory/state/persistence files
- `settings.json`: user/device/theme and app order settings.
- `curriculum.json`: grade/module/lesson graph and progress states.
- `routines.json`: routine completion with daily reset logic.
- `activity_log.json`: bridge activity events.
- `memories/session_*.json`: memory snapshots generated by `memory_manager.py`.

### Personality/prompt/config files
- `CORE_PROMPT.md`, `personality.txt`, `prompt_modules/`, and `config.py` define behavior/persona/system prompt composition.

### Media/assets that should live on SD for ESP32
- `images/`, `academy_visuals/`, `music/`, voice memo cache equivalents, themed icons/wallpapers, and whiteboard snapshots/logs.

### Backend APIs/modules that must remain off-device
- `engine.py`, `brain_client.py`, MLX model execution, speech recognition, advanced pedagogy routing, and TTS generation.
- Existing FastAPI routes in `core_bridge.py` remain authoritative; ESP32 adds narrow adapter endpoints.

## 2) Architecture plan
- See ADR: `docs/adr/0001-core-esp32-frontend-architecture.md`.
- New subproject: `core_esp32_frontend/` (ESP-IDF + LVGL).
- Companion integration: `core_backend_compat/esp32_api_extension.py` (drop-in route installer).
- Feature flags strategy (backend-side):
  - `frontend_mode=desktop` (existing face_ui + pywebview)
  - `frontend_mode=mobile_flutter` (existing Flutter)
  - `frontend_mode=esp32` (new protocol + reduced payloads)

## 3) Functionality mapping from original app to ESP32 frontend
| Original Core capability | ESP32 mapping |
|---|---|
| Circular launcher with app tiles + reorder | LVGL round launcher with tile components; reorder protocol reserved in intent API |
| Face states (`happy/listening/thinking/speaking`) | mirrored via `/esp32/state` into UI state labels and animations |
| PTT voice | touch-hold button emits `ptt_start/ptt_stop` intents to backend |
| Chat + pedagogy + memory | backend-only compute; ESP32 renders status/messages |
| Whiteboard | backend-rendered whiteboard payloads; ESP32 viewport approximation |
| Gallery/music/assets | manifest-based SD storage + on-demand fetch/caching |
| Settings/theme | local view + backend sync via `settings_patch` intent |
| Activity/diagnostics | structured logs on SD + backend events |

## 4) Step-by-step implementation
1. Bootstrap `core_esp32_frontend/` as ESP-IDF app with custom partitions and PSRAM defaults.
2. Add platform event bus component for decoupled UI/protocol/storage modules.
3. Add storage manager with SD folder auto-create, manifest bootstrap, cache entry write path, and fallback health state.
4. Add protocol client:
   - poll `/esp32/state`
   - emit app state events
   - send intents/touch interactions (scaffolded functions)
5. Add LVGL runtime with round root container, status area, face status, launcher tiles, and SD fault warning.
6. Add backend compatibility extension for `/esp32/state`, `/esp32/intent`, `/esp32/assets/manifest`.
7. Document build/flash/SD prep and protocol contract.

## 9) Final functionality parity report
### Preserved exactly
- Companion architecture where Core backend remains source of truth for AI behavior.
- Launcher-centric app navigation concept and circular identity.
- State-driven face/status UI semantics.

### Approximated on ESP32
- High-fidelity glass/blur web visuals approximated with LVGL primitives.
- Complex transition effects simplified to meet memory/frame constraints.
- Whiteboard and rich text rendering reduced to practical LVGL equivalents.

### Backend-only by design
- MLX model inference, advanced pedagogical orchestration, speech recognition, high-quality cloud TTS.
- Heavy JSON prompt composition and long-history reasoning.

### Impossible/unsafe fully on-device
- Running the Mac MLX model and full Python runtime stack natively on ESP32-S3.
- Retaining identical desktop browser-grade animation and layout fidelity at all times.
